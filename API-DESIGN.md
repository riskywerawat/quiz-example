ออกแบบ API อย่างไรไม่ให้งง? 

หนึ่งในเรื่องที่ Developer สาย Web App อย่างเรามักจะทำกันจนเป็นเรื่องปกติ คือการสร้าง API เพื่อให้ Frontend ทำการเรียกใช้งานข้อมูล หากเรามาพูดถึงเรื่อง API ไหนเป็น API ที่ดีไม่ดี อาจจะมีหลายด้านที่เราต้องพิจารณากัน ซึ่งในวันนี้เราจะมาพูดกันเรื่องของการตั้งชื่อ API ครับ 
 

ชื่อ API สำคัญอย่างไร 

Endpoint URL ที่อ่านง่ายและคาดเดาได้ เป็นส่วนที่จะช่วยให้ Dev รวมถึงคนใช้งานลดระยะเวลาในการตีความ เรียนรู้ รวมถึงอาจจะช่วยลดความผิดพลาดในการนำไปใช้งานได้ด้วย ในทางกลับกันการตั้งชื่อที่ผิดพลาดหรืออ่านยาก จะทำให้การ Maintain ระบบเป็นเรื่องที่ยุ่งยากขึ้น รวมถึงการจัดทำ Spec อาจจะเกิดความสับสนได้นั่นเอง 
 

ก่อนจะรู้จักการตั้งชื่อ ต้องรู้จัก REST API กันก่อน 

REST API เป็นคำที่ย่อมาจาก Representational State Transfer Application Programming Interface ตัวของ REST API นั้นถือเป็น Standard Protocol รูปแบบ Architectural Style ที่ถึงชื่อจะบอกว่าเป็นมาตรฐาน แต่ตัว REST นั้นกลับมีความยืดหยุ่นค่อนข้างสูง ซึ่งส่งผลให้ REST นั้นสามารถขยายระบบได้ง่าย 

หัวใจหลักของ REST คือการมองทุกอย่างเป็น "ทรัพยากร (Resource)" และเราจะกระทำกับทรัพยากรเหล่านั้นผ่าน "การกระทำมาตรฐาน" ที่กำหนดไว้แล้ว ซึ่งก็คือ HTTP Methods ที่เราคุ้นเคยกันดี คราวนี้เรามาเข้าใจความหมายของคำเหล่านี้ให้ลึกซึ้งกันอีกสักนิดกันครับ 

- Resource (ทรัพยากร): คือสิ่งใดๆ ก็ตามที่เราต้องการจัดการผ่าน API เช่น users, products, orders เป็นต้น 
- HTTP Methods (การกระทำ): คือคำสั่งที่เราใช้จัดการกับ Resource นั้นๆ 
- GET: ขอข้อมูลจาก Resource 
- POST: สร้าง Collection Resource ใหม่ 
- PUT: อัพเดททับ Individual Resource เดิม 
- PATCH: อัพเดทบางส่วนของ Individual Resource 
- Delete: ลบ Resource 

เมื่อเราเข้าใจคอนเซ็ปต์นี้แล้ว การตั้งชื่อ API ตามหลัก RESTful ก็จะกลายเป็นเรื่องที่สมเหตุสมผลและทำตามได้ง่ายขึ้นทันที 
 

Best Practices ในการตั้งชื่อ API 

อย่างที่กล่าวไปตอนต้นว่า การตั้งชื่อ REST API ไม่ได้มีหลักการตายตัวเนื่องจากเป็นมาตรฐานที่ค่อนข้างมีความยืดหยุ่น ดังนั้นสิ่งที่เราจะพูดกันต่อไปนี้เป็นเพียงการแนะนำ Best Practice เท่านั้น โดยจะมีหลักการอะไรบ้าง เราไปดูกันเลย 

1. ใช้ "คำนามพหูพจน์ (Plural Nouns)" แทนทรัพยากร 
หลักการที่สำคัญที่สุดคือ URL Endpoint ควรจะสื่อถึง "สิ่งของ" หรือ "ทรัพยากร" ไม่ใช่ "การกระทำ" เพราะการกระทำจะถูกกำหนดโดย HTTP Method อยู่แล้ว และควรใช้เป็นรูปพหูพจน์เสมอเพื่อให้สื่อถึง "ชุดของทรัพยากร" (Collection) คราวนี้มาดูตัวอย่างที่ดีและไม่ดีกันครับ

2. ใช้ HTTP Methods สำหรับการกระทำ 
ดังที่กล่าวไปข้างต้น เราไม่ต้องใส่คำกริยา (Verb) เช่น get, create, delete เข้าไปใน URL เพราะ HTTP Method ทำหน้าที่นั้นอยู่แล้ว

3. ใช้ตัวอักษรพิมพ์เล็ก (Lowercase) และเครื่องหมายขีดกลาง (-) 
เพื่อความสม่ำเสมอและหลีกเลี่ยงปัญหาจากระบบที่อาจจะ Case-Sensitive ควรใช้ตัวอักษรพิมพ์เล็กทั้งหมด และใช้เครื่องหมายขีดกลาง (-) แทนการใช้ Underscore (_) หรือ CamelCase

4. จัดการความสัมพันธ์ด้วย Nested Resources 
สำหรับทรัพยากรที่มีความสัมพันธ์กันแบบ Parent-Child เราสามารถแสดงความสัมพันธ์นั้นได้โดยตรงใน URL เพื่อให้เข้าใจง่าย ยกตัวอย่างเช่น "หากเราต้องการ "ดูคำสั่งซื้อ (orders) ทั้งหมดของลูกค้า (customer) ที่มี ID 123" เราสามารถออกแบบ API ได้เป็น "GET /customers/123/orders/" วิธีนี้จะทำให้เราเห็นภาพชัดเจนว่าเรากำลังทำงานกับ "orders" ที่อยู่ภายใต้ "customers" ที่มี ID เท่ากับ 123 นั่นเอง

5. จัดการผลลัพธ์ขั้นสูงด้วย Query Parameters (ปรับปรุงและเพิ่มเติม) 
Query Parameters คือส่วนของ URL ที่อยู่หลังเครื่องหมายคำถาม (?) ใช้สำหรับส่งข้อมูลเพิ่มเติมโดยที่ไม่กระทบต่อ Path หลักของ Resource นั้น ซึ่งโดยทั่วไปแล้วจะใช้ในการจัดการกับผลลัพธ์ต่างๆ โดยที่ไม่ต้องสร้าง Endpoint ใหม่ อาทิ การกรอง, การจัดเรียง เป็นต้น  

5.1 การกรองข้อมูล (Filtering) 
ใช้สำหรับจำกัดผลลัพธ์ให้ตรงตามเงื่อนไขที่ต้องการ เช่น 
- ดึงข้อมูลบทความ (articles) ที่มีสถานะเป็น 'published': "GET /articles?status=published" 
- ดึงข้อมูลสินค้า (products) ที่อยู่ในหมวดหมู่ 'electronics': "GET /products?category=electronics" 

5.2 การจัดเรียงข้อมูล (Sorting) 
ใช้สำหรับเรียงลำดับผลลัพธ์ตามฟิลด์ที่กำหนด โดยมักใช้เครื่องหมายบวก (+ หรือไม่ใส่) สำหรับน้อยไปมาก (Ascending) และเครื่องหมายลบ (-) สำหรับมากไปน้อย (Descending) เช่น 
- ดึงข้อมูลสินค้า (products) โดยเรียงตามราคาจากน้อยไปมาก: "GET /products?sort=price" หรือ "GET /products?sort=+price" 
- ดึงข้อมูลผู้ใช้ (users) โดยเรียงตามวันที่สร้างล่าสุดก่อน และตามด้วยชื่อ A-Z: "GET /users?sort=-created_at,first_name" 

5.3 การแบ่งหน้า (Pagination) 
ใช้สำหรับการจัดแบ่ง Resource ที่มีขนาดใหญ่ให้มีขนาดที่เล็กลง เพื่อป้องกันการส่งข้อมูลขนาดใหญ่เกินไปในครั้งเดียว และช่วยเพิ่มประสิทธิภาพของ API 
- ขอข้อมูลหน้าที 2 โดยแสดงผล 20 รายการต่อหน้า: "GET /users?page=2&limit=20" 
- อีกรูปแบบที่นิยมคือการใช้ offset (ตำแหน่งเริ่มต้น) และ limit (จำนวน): "GET /users?offset=20&limit=20 (ข้าม 20 รายการแรก และเอา 20 รายการถัดไป)" 

5.4 การเลือกฟิลด์ข้อมูล (Field Selection / Sparse Fieldsets) 
ช่วยให้ Client สามารถร้องขอเฉพาะฟิลด์ข้อมูลที่ต้องการได้ เพื่อลดขนาดของข้อมูลที่รับส่ง ทำให้ API ตอบสนองได้เร็วขึ้น 
- ดึงข้อมูลบทความ (articles) โดยเอาแค่ฟิลด์ title, author, และ published_date: "GET /articles?fields=title,author,published_date" 

5.5 การค้นหา (Searching) 
ในกรณีที่ต้องการค้นหาแบบอิสระ (Free-text search) สามารถใช้พารามิเตอร์ q (ย่อมาจาก query) ได้ 
- ค้นหาสินค้าทั้งหมดที่มีคำว่า "laptop" อยู่ในชื่อหรือคำอธิบาย: "GET /products?q=laptop" 

6. ใส่เวอร์ชันของ API ใน URL (Versioning) 
ในกรณีที่เราต้องการพัฒนาและปรับเปลี่ยน API เดิมที่เคยทำไว้อยู่แล้ว ไม่ว่าจะเป็นการเพิ่มประสิทธิภาพ การเพิ่ม Logic หรือเหตุผลต่างๆ นอกจากนี้ การทำ Versioning จะช่วยให้ผู้ใช้งานเก่าที่ยังไม่พร้อมอัปเดตยังสามารถใช้งานเวอร์ชันเดิมได้ต่อไป โดยไม่ได้รับผลกระทบจากการเปลี่ยนแปลงในเวอร์ชันใหม่ วิธีที่นิยมที่สุดคือการระบุเวอร์ชันไว้ใน Path ของ URL เช่น "https://api.example.com/v1/products", "https://api.example.com/v2/products" เป็นต้น 
 

สรุป 

การตั้งชื่อ API ที่ดีและสอดคล้องกับหลัก RESTful ไม่ใช่แค่เรื่องของความสวยงาม แต่คือการลงทุนที่คุ้มค่าในระยะยาว ซึ่งจะช่วยให้ 
- Developer Experience (DX) ดีขึ้น: Dev สามารถเข้าใจและนำไปใช้งานได้เร็วขึ้น 
- ลดภาระการทำเอกสาร: URL ที่สื่อความหมายได้ดี ลดความจำเป็นในการอธิบายซ้ำซ้อน 
- ง่ายต่อการบำรุงรักษา: โครงสร้างที่คาดเดาได้ทำให้การแก้ไขและต่อเติมในอนาคตทำได้ง่าย 
เพียงแค่เริ่มต้นจากการใช้ คำนามพหูพจน์ สำหรับทรัพยากร, ปล่อยให้ HTTP Methods ทำหน้าที่ของมัน, และใช้ Query Parameters อย่างชาญฉลาดในการจัดการผลลัพธ์ เราก็จะสามารถออกแบบ API ที่เป็นมาตรฐานสากลและเป็นมิตรกับนักพัฒนาไปอีกขั้นแล้วครับ
