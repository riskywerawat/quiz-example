จากบทความเรื่อง "เขียน Code ให้ Clean ขึ้น ด้วยเทคนิค Clean Code จากลุง Robert C. Martin" สามารถสรุปประเด็นสำคัญเป็นหัวข้อสั้นๆ ได้ดังนี้ครับ 

1. การตั้งชื่อ (Meaningful Names) 

สื่อความหมายชัดเจน: ชื่อตัวแปร ฟังก์ชัน หรือคลาส ควรบอกได้ว่าทำหน้าที่อะไร ใช้งานอย่างไร โดยไม่ต้องพึ่ง Comment 

ไม่ทำให้เข้าใจผิด (Avoid Disinformation): หลีกเลี่ยงคำที่สื่อความหมายผิด เช่น การใช้คำว่า List กับตัวแปรที่ไม่ใช่ List หรือชื่อย่อของระบบปฏิบัติการที่อาจทำให้สับสน 

แตกต่างอย่างมีความหมาย: ชื่อที่คล้ายกันต้องมีความแตกต่างที่ชัดเจน ไม่ใช่แค่เติมตัวเลขต่อท้าย (เช่น a1, a2) 

อ่านออกเสียงได้: เพื่อให้ง่ายต่อการพูดคุยสื่อสารในทีม 

ค้นหาได้ง่าย: หลีกเลี่ยงชื่อตัวอักษรเดียวสั้นๆ ที่ค้นหาใน Code ได้ยาก 

2. Objects และ Data Structures 

Objects: ควรซ่อนข้อมูล (Data) ไว้ข้างหลังและเปิดเผยเฉพาะพฤติกรรม (Function) ให้เรียกใช้งาน 

Data Structures: ทำหน้าที่เปิดเผยข้อมูลออกมาตรงๆ และไม่มีฟังก์ชันการทำงานที่ซับซ้อนเหมือน Objects 

3. การจัดการข้อผิดพลาด (Error Handling) 

ใช้ Exceptions แทน Return Codes: เพื่อให้ Code อ่านง่ายและแยกส่วน Logic กับ Error handling ออกจากกัน 

เขียน Try-Catch-Finally ก่อน: ให้มองเป็น Transaction เพื่อจัดการ State ของโปรแกรมให้สอดคล้องกัน 

ห้าม Return Null: การส่ง Null กลับมาจะทำให้ต้องมาคอยเช็ก Null ตลอดเวลา ควรใช้การ Re-throw exception หรือ Return object พิเศษแทน 

4. การเขียน Unit Tests 

กฎ 3 ข้อของ TDD: 

ไม่เขียน Production code จนกว่าจะมี Failing unit test 

เขียน Test แค่พอให้มัน Fail (Compile error ก็ถือว่า Fail) 

เขียน Production code แค่พอให้ Test ผ่าน 

Clean Tests: หัวใจสำคัญคือ "ความอ่านง่าย" (Readability) ต้องชัดเจนและเรียบง่าย 

กฎ 5 ข้อของ Test ที่ดี (F.I.R.S.T): Fast (เร็ว), Independent (เป็นอิสระต่อกัน), Repeatable (ทำซ้ำได้), Self-Validating (ตรวจสอบผลได้ด้วยตัวเอง), Timely (เขียนในช่วงเวลาที่เหมาะสม) 

5. การออกแบบ Class 

ขนาดเล็ก (Small): Class ไม่ควรมีขนาดใหญ่เกินไป 

Encapsulation: ซ่อนตัวแปรและ Utility functions เป็น private/protected 

Single Responsibility Principle (SRP): หนึ่ง Class ควรมีหน้าที่รับผิดชอบเพียงเรื่องเดียว และมีเหตุผลเดียวที่จะถูกแก้ไข
